# CI 实施优先级与路线图

## 1. 实施阶段规划

### 阶段 1: 基础语法检查 (Week 1-2) ✅ 最高优先级

**目标**: 立即获得 CI 价值，无需自托管 Runner

```yaml
# 已完成: .github/workflows/tests.yml 中的以下任务
- detect_changes    # 变更检测
- audit            # Formula 审计
- dependency_check # 依赖检查
```

**具体任务**:

| 序号 | 任务 | 工作量 | 依赖 |
|------|------|--------|------|
| 1.1 | 部署基础工作流 | 1天 | 无 |
| 1.2 | 修复现有 Formula 语法问题 | 2天 | 1.1 |
| 1.3 | 配置分支保护规则 | 0.5天 | 1.1 |
| 1.4 | 添加 PR 模板 | 0.5天 | 无 |

**预期成果**:
- PR 提交时自动运行语法检查
- 不符合规范的 Formula 无法合并
- 减少基础错误进入主分支

**GitHub 设置**:
```
Settings → Branches → Branch protection rules → main:
  ✓ Require status checks to pass before merging
  ✓ Require branches to be up to date before merging
  ✓ Status checks: audit, dependency_check
```

### 阶段 4: 全量回归测试 (Week 7-8) 📅 计划内

**目标**: 定时全量测试，发现潜在回归问题

**触发策略**:

```yaml
schedule:
  - cron: '0 2 * * 1'  # 每周一凌晨 2 点
    # 避开工作日，利用低峰期
    # 周一便于及时处理周末发现的问题
```

**容错策略**:

```ruby
# 允许部分失败，不阻塞整体流程
regression_test:
  strategy:
    fail-fast: false  # 一个失败不停止其他
  steps:
    - run: |
        # 失败超过 5 个才标记整体失败
        if [ $FAILED -gt 5 ]; then
          exit 1
        fi
```

## 3. 风险评估

### 3.1 风险矩阵

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| Runner 离线 | 中 | 高 | 设置多个 Runner，fallback 到语法检查 |
| 构建超时 | 高 | 中 | 增加超时时间，优化 Formula |
| 磁盘不足 | 中 | 高 | 自动清理，监控磁盘使用 |
| 网络问题 | 低 | 中 | 配置镜像源，本地缓存 |
| GitHub API 限制 | 低 | 低 | 使用 GITHUB_TOKEN，控制频率 |

### 3.2 应急预案

```bash
#!/bin/bash
# runner-failover.sh - Runner 故障转移脚本

# 检测 Runner 状态
RUNNER_STATUS=$(curl -s -H "Authorization: token $TOKEN" \
  https://api.github.com/repos/$REPO/actions/runners | \
  jq -r '.runners[] | select(.name | contains("loong64")) | .status')

if [ "$RUNNER_STATUS" != "online" ]; then
  echo "⚠️ Runner 离线，发送通知"
  # 发送邮件/Slack 通知
  
  echo "切换 CI 模式为仅语法检查"
  # 临时修改工作流，跳过 build_test
fi
```
